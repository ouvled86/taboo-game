# 03. WordPress Configuration

The configuration of the `wordpress` service follows modern DevOps practices, prioritizing flexibility and security by separating the configuration from the application code. The entire setup is driven by environment variables, which are consumed by a setup script to generate the final configuration files.

### The Source of Truth: The `.env` File

Unlike a traditional setup where credentials might be hard-coded, this project defines all configuration parameters in the `.env` file. The `docker-compose.yml` file injects these variables into the WordPress container's environment at runtime.

```yaml
# docker-compose.yml
services:
  wordpress:
    # ...
    env_file:
      - .env
    # ...
```

This file contains all the critical and user-specific values, such as:
- MariaDB database names, users, and passwords (`MYSQL_DATABASE`, `MYSQL_USER`, etc.).
- WordPress administrator credentials (`WP_ADMIN_USER`, `WP_ADMIN_PASSWORD`).
- Redis connection details (`REDIS_HOST`, `REDIS_PORT`).
- The domain name for the site (`DOMAIN_NAME`).

This approach is powerful because the WordPress image itself contains no sensitive information and can be configured for any environment (development, staging, production) simply by providing a different `.env` file.

### The Configuration Engine: `setup.sh`

The `tools/setup.sh` script is the engine that transforms the environment variables into a working WordPress configuration. When the container starts for the first time, this script reads the environment variables and uses the `wp-cli` tool to:
- **Create `wp-config.php`**: This is the main configuration file for WordPress. The script uses `wp config create` to generate it.
  ```bash
  wp config create --dbname=${MYSQL_DATABASE} --dbuser=${MYSQL_USER} --dbpass=${MYSQL_PASSWORD} --dbhost=mariadb ...
  ```
- **Set Additional Constants**: It also uses `wp config set` to add bonus-related configurations, like the Redis host, directly into `wp-config.php`.
  ```bash
  wp config set WP_REDIS_HOST ${REDIS_HOST} ...
  ```

### The Result: `wp-config.php`

The final `wp-config.php` file, located at `/var/www/html/wp-config.php`, is the bridge between the WordPress PHP application and its environment. It's a PHP file that defines a series of constants, including:
- **Database Connection Settings:** `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_HOST`.
- **Authentication Keys and Salts:** A set of unique, random security keys used to strengthen user authentication and cookies. These are automatically generated by `wp-cli`.
- **Redis Connection Settings:** `WP_REDIS_HOST` and `WP_REDIS_PORT`.
- **Table Prefix:** `\$table_prefix`.

Because this file is created inside the `/var/www/html` directory, which is a persistent Docker volume, it survives container restarts. The `setup.sh` script checks for the existence of this file (`if [ ! -f "/var/www/html/wp-config.php" ]`) to ensure that the entire installation and configuration process only runs once.

### PHP Process Configuration: `www.conf`

At a lower level, the PHP-FPM process itself is configured by `srcs/requirements/wordpress/conf/www.conf`. This file is not about the WordPress *application*, but about the PHP *server*. Its key directives are:
- **`listen = 0.0.0.0:9000`**: Tells PHP-FPM to listen for FastCGI requests from any network interface on port 9000. This is how it accepts connections from Nginx.
- **`user = www-data` / `group = www-data`**: Sets the user and group that the PHP processes will run as. This is a security measure to ensure the process doesn't run with elevated (`root`) privileges.
- **`pm = dynamic`**: Configures the process manager to scale the number of PHP worker processes up and down based on demand, which is efficient for managing server resources.
